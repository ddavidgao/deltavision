### BUG-0008 — DVObservation constructed without consumed_frame_size silently zero-divides cost-split

`severity:minor` `status:open` `discovered:2026-04-26` `#observer` `#cost-split` `#api`

**Discovered during:** Reviewing observer.py during the v1.0.7-dev cost-split rename: noticed that dv_internal_tokens() returns 0 when consumed_frame_size is None

**Symptom**

If user code constructs a DVObservation directly (bypassing DeltaVisionObserver) without setting consumed_frame_size, dv_internal_tokens() returns 0. Any code that computes savings as 1 - model_facing/dv_internal then divides by zero or silently produces 100%/inf savings — a critical-looking number with no underlying truth.

**Reproduction**

from observer import DVObservation\nimport PIL.Image\nobs = DVObservation(obs_type='full_frame', step=0, trigger='x', diff_ratio=0, phash_distance=0, anchor_score=0, action_had_effect=False, frame=PIL.Image.new('RGB', (100,100)))\nprint(obs.dv_internal_tokens())  # 0 — consumed_frame_size was never set\n# now any 'savings = 1 - mf/internal' computation explodes

**Root cause**

consumed_frame_size is an Optional dataclass field, defaulting to None. The observer always populates it, but a DVObservation built outside the observer (mocks, custom integrations) won't. dv_internal_tokens() returns 0 in this case rather than raising. Choosing 'silently return 0' lets back-compat code paths work but masks errors in cost-split callers.

**Notes**

Two reasonable fixes: (1) Make consumed_frame_size required at the dataclass level, force callers to provide it, break code that doesn't. (2) Have dv_internal_tokens() raise if consumed_frame_size is None instead of returning 0 — explicit failure beats silent zero-division. Defer until we ship the cost-split publicly (v1.0.7); whichever fix we pick should land in the same release that makes model_facing_tokens()/dv_internal_tokens() the documented public API.

### BUG-0006 — Greedy bbox-merge runs before MAX_REGIONS truncation, regressing scripted-77

`severity:major` `status:deferred` `discovered:2026-04-25` `introduced-in:ff53ade` `#classifier` `#vision` `#benchmark` `#ci` `#v1.0.6`

**Discovered during:** v1.0.6 release — benchmark-reproducibility CI (`scripted-77`) failed: expected 77.2% savings, measured 72.1%

**Symptom**

Greedy bbox-merge optimizer (BUG: ff53ade) reduces measured savings on the scripted spreadsheet benchmark from 77.2% → 72.1% when enabled by default. Same observer state, same trace; toggling the flag flips the result deterministically.

**Reproduction**

1. `python examples/spreadsheet_observation_cost.py` on HEAD: 72.1% savings, 9514 DV / 34125 FF tokens.
2. Edit config.py line 61: `BBOX_MERGE_ENABLED: bool = False`.
3. Re-run: 77.2% savings, 7780 DV / 34125 FF tokens.
4. CI: github.com/ddavidgao/deltavision/actions, `scripted-77` job, asserts `abs(pct - 77.2) < 1.0`.

**Root cause**

vision/diff.py runs `merge_bboxes_for_min_cost(bboxes)` on the full unmerged contour list (line 145), THEN sorts by area and truncates to `bboxes[: config.MAX_REGIONS]` (line 160). The merger optimizes 'send all bboxes, minimize cost', but the actual pipeline is 'send top-N largest, minimize cost'. When the unmerged path would have dropped 2+ small bboxes for free via truncation, merging them into the top-N just adds their area back to the cost.

**Notes**

User accepts savings drift on this benchmark — bbox-merge is non-deterministic w.r.t. input contour patterns, and 5pp drift on a single benchmark is within noise. The greedy merger still wins on multi-step DV-MCP runs (38.1% → 48.9% across 16 saved traces). Loosening the scripted-77 CI assertion would be the right move; the merge-after-truncate hypothesis is logged here for future revisit. Won't block v1.0.6.

### BUG-0001 — deltavision==1.0.2 wheel had no `import deltavision` namespace

`severity:blocker` `status:fixed` `discovered:2026-04-12` `fixed-in:v1.0.3` `#packaging` `#v1.0.x`

**Discovered during:** Cold-install dogfood test of v1.0.2 from a fresh venv

**Symptom**

`pip install deltavision==1.0.2 && python -c 'import deltavision'` raised ModuleNotFoundError on a fresh venv.

**Reproduction**

Pre-fix: `pip install deltavision==1.0.2; python -c 'import deltavision'` from any directory not containing a local `vision/` raised ModuleNotFoundError. Post-fix: same command succeeds against 1.0.3+.

**Root cause**

The wheel installed flat modules (observer.py, vision/, agent/) at site-packages root with no umbrella `deltavision/` package, so `import deltavision` failed. A local CWD with `vision/` or `observer.py` would also shadow the installed copy.

**Fix**

`v1.0.3` — Ship a deltavision/ umbrella package alongside the flat modules so `import deltavision` works regardless of CWD

**Notes**

This is the canonical example for the 'test like a naive user, not an insider' rule in ~/.claude/CLAUDE.md. The insider-style `from observer import ...` tests were all green; only a cold-pip-install caught it.

### BUG-0002 — Map-pan misclassified as NEW_PAGE by raw diff_ratio classifier

`severity:major` `status:fixed` `discovered:2026-04-22` `fixed-in:0f03b7f` `#classifier` `#vision`

**Discovered during:** SF mapsheets DV trace inspection — agent appeared to misclassify Maps pans as full new pages, billing 1365 tokens per pan

**Symptom**

On Google Maps, panning the viewport a few hundred pixels triggered NEW_PAGE classification and a full-frame send, even though the underlying scene was unchanged.

**Reproduction**

Pan a Google Maps frame ~200px in any direction. Pre-fix: classifier returns NEW_PAGE because diff_ratio > new_page threshold even though the underlying scene is identical. Post-fix: ORB-warped re-diff drops diff_ratio under threshold and classifies DELTA.

**Root cause**

The original classifier used raw pixel-level diff_ratio. A camera/viewport pan shifts every visible pixel even when the underlying scene is identical, so diff_ratio crosses the new_page threshold for cosmetic reasons.

**Fix**

`0f03b7f` — Residual-first warp using ORB feature matching + RANSAC homography, then re-diff on the warped image; pans now classified as DELTA, not NEW_PAGE

**Notes**

Cost the SF benchmark a full ~5pp of measured savings before the fix. Validated by replaying run_20 trace through the new classifier and watching every pan flip from new_page→delta.

### BUG-0003 — Proxy could bill more than a full frame on fragmented diffs

`severity:major` `status:fixed` `discovered:2026-04-23` `fixed-in:0f03b7f` `#proxy` `#cost-model`

**Discovered during:** DV proxy log inspection on a SF run — some steps billed dv_cost > 1365 (FF cost), violating the 'always cheaper than full frame' invariant

**Symptom**

On steps with fragmented diffs (many small scattered changes), the proxy emitted dv_tokens > FULL_FRAME_TOKENS, making DV strictly more expensive than FF on that step.

**Reproduction**

Construct a 1280×800 frame pair where the second has many small (40×40) scattered black regions. Diff produces N>=8 separate bboxes, each costing 85 base tokens + tile cost. Sum exceeds 1365. Pre-fix: dv_cost = sum. Post-fix: dv_cost = 1365 with trigger='token_cap_fallback'.

**Root cause**

Each crop region carries an 85-token base overhead (CROP_BASE_TOKENS). On steps with many small scattered changes, the sum of base+tile costs across N regions exceeded the 1365 cost of sending one full frame, making DV strictly worse than FF on that step.

**Fix**

`0f03b7f` — Cap dv_cost at FULL_FRAME_TOKENS in DVState.process_screenshot; if crops would cost more, fall back to a full frame instead

**Notes**

Now guarded by tests/test_proxy_token_cap.py::test_token_cap_on_fragmented_diff — the test enforces dv_cost ≤ FULL_FRAME_TOKENS for any frame pair the proxy processes.

### BUG-0004 — Agent loses spatial context after long sequences of small-delta observations

`severity:major` `status:fixed` `discovered:2026-04-24` `fixed-in:6f47491` `#proxy` `#agent-loop` `#ux`

**Discovered during:** Watching the SF mapsheets DV agent loop — it got stuck in a Google Sheets formatting dialog, took 49 steps to complete what FF did in 45

**Symptom**

Agent took more steps than FF on tasks involving sustained dialog/menu interactions (Sheets conditional formatting, color pickers). DV won on per-step tokens but lost on total tokens because of the step-count blow-up.

**Reproduction**

Run any agent on a multi-step dialog interaction (Sheets conditional formatting, color picker, etc.). Pre-fix: the agent makes 8+ tiny edits, each a small-crop delta, and gets confused about overall page state. Post-fix: a full frame is forced every 5 deltas, breaking the spatial-confusion spiral.

**Root cause**

The CV pipeline correctly emitted small deltas during multi-step dialog interactions, but the agent's working memory of the full page state degraded across consecutive crop-only observations. By the 8th–10th delta the agent was 'reasoning blindly' about layout it could no longer see.

**Fix**

`6f47491` — Force a full-frame response after DV_DELTA_REFRESH_EVERY=5 consecutive deltas, so the agent re-anchors instead of reasoning from a long sequence of tiny crops

**Notes**

Combined with a leaner screenshot prompt this dropped agent step count from 49 → 32 on the SF mapsheets task (vs FF's 45 steps). Validated by tests/test_proxy_token_cap.py::test_periodic_refresh_fires_after_n_deltas. The 53.2% total savings number on the v1.0.6 demo video depends on this.

### BUG-0005 — Phantom-state DV frames inflated early-step savings on demo videos

`severity:minor` `status:fixed` `discovered:2026-04-24` `fixed-in:1da3e46` `#video` `#benchmark` `#data-quality`

**Discovered during:** Building the v1 ffvsdv comparison video — DV starting card showed pre-filled spreadsheet rows that FF didn't have, breaking the 'same starting state' claim

**Symptom**

DV side of side-by-side comparison videos showed sheet pre-filled with stale data while FF side started empty, exaggerating early-step DV efficiency.

**Reproduction**

Look at any DV-vs-FF SBS frame from a run where DV started with phantom data: the DV pane shows pre-filled cells while the FF pane is empty.

**Root cause**

The DV browser session reused a Chrome profile that had stored Sheets state from a prior session, so the first DV screenshot already had data. The FF browser was a fresh profile.

**Fix**

`1da3e46` — Per-run DV_EXCLUDE_STEPS hand-cull list; recompute dv_cumulative over kept steps so SBS card matches outro headline

**Notes**

The fix is run-specific (a hand-curated set of step indexes per saved trace) rather than a generic guard. Acceptable for video rendering but would be the wrong shape for production pipelines. If we ever re-record a trace from a known-clean state, the cull list goes empty (as it does for run_24).
